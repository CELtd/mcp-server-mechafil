# MCP Tools Context & Best Practices

## Available Tools

### `get_historical_data`
- **Endpoint**: `/historical-data`
- **Purpose**: Retrieve cached on-chain Filecoin network data. Use this to establish current conditions, describe historical trends, and validate simulation assumptions against real data.
- **Arguments**:
  - `fields` (optional): string or list of field names to filter the response (use for plots or targeted queries).
- **Response**: JSON string with a `data` dictionary. Fields fall into five distinct groups — each group has its own date anchor and must be treated differently.

#### Critical: Two Date Anchors

**All arrays in this response are Monday-sampled.** You must know which start date anchors each field to correctly label time axes or plot data.

| Anchor key | Value | Which fields use it |
|---|---|---|
| `hist_window_start_date` | ~180 days ago | `raw_byte_power`, `renewal_rate`, `filplus_rate` (and their aliases) |
| `data_start_date` | 2022-10-10 | All other arrays: power totals, supply stocks, onboarding/renewal flows, scheduled expiry and pledge release vectors |

**Rule for `start_date_key` in plot specs:**
- Use `hist_window_start_date` for: `raw_byte_power`, `renewal_rate`, `filplus_rate`
- Use `data_start_date` for: everything else (all power, supply, onboarding, renewal arrays)

#### Group 1 — Scalar summaries (no date, use as current-state defaults)

| Field | Type | Unit | Description |
|---|---|---|---|
| `raw_byte_power_averaged_over_previous_30days` | scalar | PiB/day | Median onboarding rate over the last 30 days. Use as default `rbp` input for simulations. |
| `renewal_rate_averaged_over_previous_30days` | scalar | fraction | Median renewal rate over the last 30 days. Use as default `rr`. |
| `filplus_rate_averaged_over_previous_30days` | scalar | fraction | Median FIL+ share over the last 30 days. Use as default `fpr`. |

#### Group 2 — Short-window rate arrays (~180 days, anchored to `hist_window_start_date`)

These cover only the recent window and are suitable for trend analysis and simulation seeding.

| Field | Type | Unit | Description |
|---|---|---|---|
| `raw_byte_power` | flow | PiB/day | Daily RBP **onboarding rate** — NOT total power. Preferred alias: `raw_byte_power_onboarded_pib_per_day`. |
| `renewal_rate` | rate | fraction | Fraction of expiring power renewed each day. |
| `filplus_rate` | rate | fraction | Fraction of daily onboarding that is FIL+ verified. |

**Common pitfall**: `raw_byte_power` is a rate (PiB/day), not the total network size. Never describe it as “network power”. For total power use `historical_raw_power_eib`.

#### Group 3 — Full-history arrays (back to 2022-10-10, anchored to `data_start_date`)

These cover the entire on-chain record available in the cache.

**Cumulative stocks** (must be differenced to get daily values — same as simulation cumstocks):

| Field | Type | Unit | Simulation equivalent | Description |
|---|---|---|---|---|
| `mined_fil` | cumstock | FIL | `cum_network_reward` | Cumulative on-chain mined block rewards from genesis. Diff to get daily minted rate. |
| `burnt_fil` | cumstock | FIL | `network_gas_burn` | Cumulative on-chain burnt FIL (gas + penalties). Diff to get daily burn rate. |

**Instantaneous stocks** (read directly as the value on that day):

| Field | Type | Unit | Simulation equivalent | Description |
|---|---|---|---|---|
| `locked_fil` | stock | FIL | `network_locked` | Total locked FIL (pledge + unvested rewards) on that day. |
| `circ_supply` | stock | FIL | `circ_supply` | On-chain circulating supply on that day. |
| `historical_raw_power_eib` | stock | EiB | `network_RBP_EIB` | Total network RBP. Alias: `network_raw_power_eib`. |
| `historical_qa_power_eib` | stock | EiB | `network_QAP_EIB` | Total network QAP. Alias: `network_qa_power_eib`. |

**Flow arrays** (read directly as daily rates):

| Field | Type | Unit | Description |
|---|---|---|---|
| `historical_onboarded_rb_power_pib` | flow | PiB/day | Daily RBP onboarded over the full history. Same concept as `raw_byte_power` but spanning back to 2022-10-10. Use this for long-term onboarding trend analysis. |
| `historical_onboarded_qa_power_pib` | flow | PiB/day | Daily QAP onboarded over the full history. |
| `historical_renewed_rb_power_pib` | flow | PiB/day | Daily RBP renewed over the full history. |
| `historical_renewed_qa_power_pib` | flow | PiB/day | Daily QAP renewed over the full history. |

**Note on `mined_fil` and `burnt_fil`**: to get a daily mined/burnt rate from the Monday-sampled arrays, apply `(value[t] - value[t-1]) / 7`.

#### Group 4 — Simulation seed scalars (current-state initializers; rarely needed for analysis)

These are scalar values used to initialize the mechafil-jax simulation at `data_start_date`. They are present in the response but are not intended for direct narrative use.

| Field | Unit | Description |
|---|---|---|
| `rb_power_zero` | **PiB** | Total RBP at `current_date` used to seed the simulation. Note: unit is PiB (not EiB), despite no unit suffix in the name. |
| `qa_power_zero` | **PiB** | Total QAP at `current_date` in PiB. |
| `circ_supply_zero` | FIL | Circulating supply at `data_start_date`. |
| `locked_fil_zero` | FIL | Locked FIL at `data_start_date`. |
| `start_vested_amt` | FIL | Total FIL vested by `data_start_date` (simulation vesting baseline). |
| `zero_cum_capped_power_eib` | EiB·days | Cumulative capped power at `data_start_date` (baseline minting offset). |
| `init_baseline_eib` | EiB | Baseline power function value at `data_start_date`. |
| `daily_burnt_fil` | FIL/day | **Historical mean** daily gas burn (average of daily increments over the full history). Used as the forward burn-rate assumption in simulation. Not the current daily rate. |
| `locked_reward_zero` | FIL | Estimated unvested reward-locked FIL at `data_start_date`. |

#### Group 5 — Simulation seed vectors (forward-looking internal vectors; some useful for analysis)

| Field | Unit | Window | Description |
|---|---|---|---|
| `rb_known_scheduled_expire_vec` | PiB/day | Forecast period from `current_date` | Scheduled RBP sector expirations per day. Useful for understanding near-term expiry pressure. |
| `qa_known_scheduled_expire_vec` | PiB/day | Forecast period from `current_date` | Scheduled QAP sector expirations per day. |
| `known_scheduled_pledge_release_full_vec` | FIL/day | Historical + forecast | Scheduled pledge collateral releases per day. |
| `burnt_fil_vec` | FIL (cumulative) | `data_start_date` to `current_date` | Cumulative burnt FIL time series (same as `burnt_fil` but used as simulation seed). Not for direct analysis — use `burnt_fil` instead. |
| `historical_renewal_rate` | fraction | `data_start_date` to `current_date` | Historical renewal rate over the full period. Same concept as `renewal_rate` but spanning the full history window. |

#### Historical ↔ Simulation field equivalence

When comparing historical data to simulation outputs, use these correspondences:

| Historical field | Simulation equivalent | Note |
|---|---|---|
| `historical_raw_power_eib` | `network_RBP_EIB` | Historical is on-chain; simulation is model output |
| `historical_qa_power_eib` | `network_QAP_EIB` | Same |
| `locked_fil` | `network_locked` | Same |
| `circ_supply` | `circ_supply` | Same field name, different source |
| `mined_fil` | `cum_network_reward` | Both cumulative; model vs on-chain |
| `burnt_fil` | `network_gas_burn` | Both cumulative; model vs on-chain |
| `historical_onboarded_rb_power_pib` | `rb_day_onboarded_power_pib` | Historical segment of the simulation array |
| `historical_onboarded_qa_power_pib` | `qa_day_onboarded_power_pib` | Same |

#### Ambiguity rules
- “raw byte power” or “network power” → use `historical_raw_power_eib` (EiB, total power)
- “onboarding rate” → use `raw_byte_power` (PiB/day) for recent 180-day trend, or `historical_onboarded_rb_power_pib` for the full history
- If unclear, ask: “Do you mean total network power (EiB) or the daily onboarding rate (PiB/day)?”
- Never describe `raw_byte_power` as “network power” — it is a rate, not a total.

## Hard Rules — Never Violate These

**Rule 1 — Baseline comparisons require a simulate call.**
`network_baseline_EIB` does not exist in historical data. If you have not called `simulate` with `requested_metric: “network_baseline_EIB”` in this session, you do not know the baseline value. Do not state, estimate, or imply whether the network is above or below the baseline without that call. The baseline is ~119–130 EiB as of early 2026 — far above network QAP (~17–23 EiB) — but always fetch the precise value rather than relying on this approximation.

**Rule 2 — Simple minting share reveals network position.**
If `daily_simple_reward` is ~80%+ of total daily rewards, the network is **below** the baseline (network_time accumulates slowly, baseline minting suppressed). Do not claim the network is above the baseline while simple minting dominates — that is a contradiction.

**Rule 3 — Never compute daily rates from raw cumulative arrays.**
Cumulative simulation fields (`cum_simple_reward`, `cum_baseline_reward`, etc.) are auto-converted to daily rates by the server. Read the returned `daily_simple_reward` / `daily_baseline_reward` arrays directly. Do not apply any further differencing or division.

### `provide_plot`
- **Purpose**: Produce a strict chart JSON object the UI can render.
- **When to use**:
  - Use only if you are not confident you can emit a valid chart JSON yourself.
  - Use when the user explicitly requests separate charts for multiple series (call once per chart).
  - If you can emit the chart JSON directly, do **not** call this tool.
- **Arguments**:
  - `series`: single field name (string), a series descriptor, or a list of series for a combined chart.
  - `start_date_key`: `data_start_date` for totals, `hist_window_start_date` for onboarding-rate series.
  - `title`/`description` (optional): chart text metadata.
- **Response**: `{ "chart": { ... } }`
- **Separate charts**:
  - If the user asks for separate charts, call `provide_plot` once per series (one chart per call).
  - If the user asks for a comparison, include multiple series in a single `provide_plot` call.

### `simulate`
- **Endpoint**: `/simulate`
- **Purpose**: Run MechaFil economic forecasts under specific scenarios and return Monday-sampled time series.
- **Key Arguments** (all optional, defaults derived from recent medians):
  - `rbp`: Raw Byte Power onboarding (PiB/day).
  - `rr`: Renewal rate (0–1 fraction).
  - `fpr`: FIL+ share (0–1 fraction).
  - `lock_target`: Target consensus pledge ratio (default 0.3).
  - `forecast_length_days`: Forecast horizon in days (critical; must match user context).
  - `sector_duration_days`: Average sector lifetime (default 540).
  - `requested_metric`: Specific output metric to return (default `"1y_sector_roi"`). Examples include `"available_supply"`, `"network_RBP_EIB"`, `"network_QAP_EIB"`, `"day_network_reward"`, `"day_pledge_per_QAP"`, `"network_baseline_EIB"`.
- **Response**: Dictionary with the requested metric array (Monday values) and an `Explanation` summarizing the actual inputs after defaults were applied.
- **MANDATORY — baseline comparisons**: `network_baseline_EIB` is **not available in historical data**. It only exists in simulation output. Whenever you need to compare network QAP/RBP to the baseline, or determine whether the network is above or below the baseline target, you **must** call `simulate` with `requested_metric: "network_baseline_EIB"`. Never estimate or guess the baseline value from historical fields — the baseline is ~119–130 EiB as of early 2026, far above network QAP (~17–23 EiB).
- **Output Interpretation (units + type)**:
  - **Power (instantaneous stock, EiB)**: `network_RBP_EIB`, `network_QAP_EIB`, `network_baseline_EIB` → total power on that day.
  - **Capped power**: `capped_power_EIB` → `min(network_power, baseline_function)` in EiB. This is the effective power that counts toward baseline minting, NOT total network power. Use `network_RBP_EIB` or `network_QAP_EIB` for actual network size.
  - **Supply (instantaneous stock, FIL)**: `available_supply`, `circ_supply`, `network_locked`, `network_locked_pledge`, `network_locked_reward` → FIL totals on that day.
  - **Cumulative stocks (FIL)**: `cum_simple_reward`, `cum_baseline_reward`, `cum_network_reward`, `total_vest`, `network_gas_burn` → running totals from Filecoin genesis. Cannot be read as "today's" value. Must be differenced to get daily values (see "Cumulative Fields" section below).
  - **Daily flows (FIL/day)**: `day_network_reward`, `day_rewards_per_sector`, `total_day_vest` → readable directly as the value for that day.
  - **Initial pledge cost**: `day_pledge_per_QAP` → FIL required per TiB of QAP for sectors onboarded that day. This is a per-unit capital cost, not a flow.
  - **ROI**: `1y_sector_roi` is a dimensionless ratio (0.15 = 15% annual ROI). Its array is 90 entries shorter than `forecast_length_days` — do not align its last entry with the forecast end date.
  - **Time axis**: arrays are Monday-sampled; consecutive values are 7 days apart.
- **Usage Patterns**:
  - Single-run forecasts for direct projections.
  - Baseline plus scenario comparisons to evaluate parameter sensitivity.
  - Metric-specific investigations (ROI, supply, rewards, pledge, power).

## Tooling Best Practices
- **Time Horizon Discipline**: Always infer or elicit user timeframes and set `forecast_length_days` explicitly; avoid relying on the 10-year default.
- **Common Mappings**: 30d≈1 month, 90d≈3 months, 180d≈6 months, 365d≈1 year, 730d≈2 years, 1825d≈5 years, 3650d≈10 years.
- **Baseline Establishment**: Start complex analyses with `get_historical_data` to anchor parameters and explain current state before simulating futures.
- **Parameter Validation**: Flag unrealistic inputs (e.g., renewal rates outside 0–1, extreme RBP values) and discuss implications before executing.
- **Metric Precision**: Use the exact metric identifiers returned by the API (e.g., `"network_QAP_EIB"` instead of legacy names) to avoid call failures.
  - If you are unsure of the exact simulation field name, ask the user to choose from a short list instead of guessing.
- **Scenario Consistency**: Keep non-varied parameters aligned across comparative runs so differences isolate the scenario factor.
- **Latency Awareness**: Responses are Monday-sampled; clarify this cadence when interpreting trends or aggregating results.
- **Explanation Field**: Cross-check the `Explanation` payload to confirm the simulation used intended defaults or overrides, and reference it when reporting inputs.
- **Post-Processing**: Translate numeric outputs into meaningful economic insights—compare start/end values, compute deltas, or express percentages where helpful—while staying strictly grounded in the documented Filecoin context and observable data.
- **Error Handling**: If tool responses indicate connection issues or missing data, inform the user, recommend reruns, or suggest checking the mechafil server status.
- **Chart Output Spec (for “plot/chart/graph” requests)**:
  - When the user asks to plot historical data, return a standalone JSON object with a `chart` key so the UI can render it. Do not prefix it with labels like “Chart Specification” and do not wrap it in code fences.
  - **Mandatory**: For any plot request, call `get_historical_data` with a `fields` filter so only the requested series are returned (single string for one series, list for comparisons). Do not call `get_historical_data` without `fields` when the user asked to plot.
  - If you already called `get_historical_data` without `fields` for a plot request, re-call it with the correct `fields` before responding.
  - If you cannot choose a single field confidently, ask the user to clarify before calling the tool.
  - Format:
    ```
    {
      "chart": {
        "series": "historical_raw_power_eib",
        "label": "Network RBP",
        "unit": "EiB",
        "start_date_key": "data_start_date"
      }
    }
    ```
  - `series` must be one field name from `get_historical_data` (e.g., `historical_raw_power_eib`, `historical_qa_power_eib`, `raw_byte_power`).
  - `start_date_key` must be `data_start_date` for totals, or `hist_window_start_date` for onboarding-rate series.
  - Do not include multiple series unless the user explicitly asked for a comparison (e.g., “plot RBP and QAP”).
  - For multi-series requests, set `series` to a list of objects:
    ```
    {
      "chart": {
        "series": [
          {"name": "historical_raw_power_eib", "label": "Network RBP", "unit": "EiB"},
          {"name": "historical_qa_power_eib", "label": "Network QAP", "unit": "EiB"}
        ],
        "start_date_key": "data_start_date"
      }
    }
    ```
  - **Response contract**: For plot requests, your final response must end with the standalone chart JSON object. Do not omit it or replace it with narrative text.
  - If you are unsure which series is requested, ask the user to clarify before plotting.
- **Note on aliases**: `raw_byte_power_onboarded_pib_per_day` = `raw_byte_power` (preferred). `raw_byte_power_onboarded_eib_per_day` is a deprecated alias with a misleading name — unit is PiB/day, not EiB/day. `network_raw_power_eib` = `historical_raw_power_eib`. `network_qa_power_eib` = `historical_qa_power_eib`. `field_meta` contains per-field type/unit/meaning for key fields and is always included.

## Cumulative Fields: How to Derive Daily Values

Several simulation outputs are **cumulative stocks** — running totals accumulated from Filecoin network genesis (Oct 15, 2020). They grow monotonically and cannot be read directly as "today's value."

**Cumulative simulation fields:**
- `cum_simple_reward` — total FIL ever minted by simple minting (FIL)
- `cum_baseline_reward` — total FIL ever minted by baseline minting (FIL)
- `cum_network_reward` — `cum_simple + cum_baseline`; used internally to compute `circ_supply` (FIL)
- `cum_capped_power_EIB` — integral of capped power over time (EiB·days); feeds `network_time`. Rarely needed directly.
- `total_vest` — cumulative total FIL vested from all sources since genesis (FIL)
- `network_gas_burn` — cumulative FIL burned by gas fees and penalties since genesis (FIL)

**To get the daily contribution from a cumulative field:**
Because `/simulate` returns Monday-sampled arrays, consecutive values are 7 days apart. To get a daily average over each 7-day window:
```
daily_avg[t] = (cum[t] - cum[t-1]) / 7    # FIL/day average over that Monday-to-Monday window
```

**Minting breakdown — simple vs baseline daily rewards:**
The MCP server automatically converts cumulative minting fields to daily rates. Use these `requested_metric` values and read the returned arrays directly — they are already FIL/day, **do NOT difference them again**:
- `cum_simple_reward`   → returned as **`daily_simple_reward`** (FIL/day)
- `cum_baseline_reward` → returned as **`daily_baseline_reward`** (FIL/day)
- `cum_network_reward`  → returned as **`daily_network_reward_cumderived`** (FIL/day, but prefer `day_network_reward`)
- `total_vest`          → returned as **`daily_vest`** (FIL/day)
- `network_gas_burn`    → returned as **`daily_gas_burn`** (FIL/day)

The Explanation field confirms the conversion. Cross-check: `daily_simple_reward[t] + daily_baseline_reward[t]` should ≈ `day_network_reward[t]`.

**`day_network_reward` is the only field that was always a pre-computed daily flow.** All others above were cumulative and are now auto-converted by the server.

## Simulation Field Reference: All 44 Output Fields

**Type legend:**
- **flow** — value applies to that specific day (read directly)
- **stock** — instantaneous total as of that day (read directly)
- **cumstock** — running total from genesis (must be differenced to get daily values)
- **const** — same value every day; treat as a scalar

### Power fields

| Field | Type | Unit | Description |
|---|---|---|---|
| `network_RBP_EIB` | stock | EiB | Total raw byte power on the network. **Preferred** over `rb_total_power_eib` (identical array). |
| `network_QAP_EIB` | stock | EiB | Total quality-adjusted power. **Preferred** over `qa_total_power_eib` (identical array). |
| `rb_total_power_eib` | stock | EiB | Duplicate of `network_RBP_EIB`. Use `network_RBP_EIB` instead. |
| `qa_total_power_eib` | stock | EiB | Duplicate of `network_QAP_EIB`. Use `network_QAP_EIB` instead. |
| `network_baseline_EIB` | stock | EiB | Exponential minting baseline: ~2.766 EiB × 2^(t/year). Use this to understand where the network stands relative to the target. When `network_QAP_EIB < network_baseline_EIB`, baseline minting is slowed. |
| `capped_power_EIB` | stock | EiB | `min(network_power, network_baseline_EIB)` — effective power for baseline minting. NOT total network power. |
| `cum_capped_power_EIB` | cumstock | EiB·days | Integral of `capped_power_EIB` over time. Used to derive `network_time`. Rarely needed directly. |
| `network_time` | stock | days-equiv | Compressed time used in the baseline minting formula. Technical internal quantity. |

| `qa_day_renewed_power_pib` | flow | PiB/day | Daily QAP renewed. **Preferred** over `day_renewed_power_QAP_PIB` (identical array). |
| `day_renewed_power_QAP_PIB` | flow | PiB/day | Duplicate of `qa_day_renewed_power_pib`. Use `qa_day_renewed_power_pib` instead. |
| `rb_sched_expire_power_pib` | flow | PiB/day | Scheduled RBP sector expirations. **Forecast-only** — starts at current_date, not historical start. |

| `full_renewal_rate` | stock | fraction | Historical renewal rate concatenated with forecast renewal rate. Full trajectory of `rr`. |

### Minting fields

| Field | Type | Unit | Description |
|---|---|---|---|
| `day_network_reward` | flow | FIL/day | Total daily block rewards (simple + baseline combined). Read directly. |
| `cum_simple_reward` → **`daily_simple_reward`** | auto-converted | FIL/day | Request `cum_simple_reward`; server returns `daily_simple_reward` with pre-differenced daily values. Read directly. |
| `cum_baseline_reward` → **`daily_baseline_reward`** | auto-converted | FIL/day | Request `cum_baseline_reward`; server returns `daily_baseline_reward`. Read directly. |
| `cum_network_reward` → **`daily_network_reward_cumderived`** | auto-converted | FIL/day | Auto-converted if requested; prefer `day_network_reward` instead. |

### Supply and locking fields

| Field | Type | Unit | Description |
|---|---|---|---|
| `circ_supply` | stock | FIL | Circulating supply = vested + minted − burned − locked. |
| `available_supply` | stock | FIL | `circ_supply + network_locked` (total supply before subtracting locked FIL). |
| `network_locked` | stock | FIL | Total locked FIL = `network_locked_pledge + network_locked_reward`. |
| `network_locked_pledge` | stock | FIL | Pledge collateral component of locked FIL only. |
| `network_locked_reward` | stock | FIL | Unvested block reward component of locked FIL (180-day linear vesting). |
| `network_gas_burn` | cumstock | FIL | Cumulative FIL burned by gas fees and penalties since genesis. NOT a daily flow — diff to get daily burn rate. |
| `disbursed_reserve` | const | FIL | ~17.07M FIL disbursed from the Mining Reserve fund. Same value every day — treat as a scalar, not a time series. |

### Pledge and reward fields

| Field | Type | Unit | Description |
|---|---|---|---|
| `day_locked_pledge` | flow | FIL/day | New pledge locked for sectors **onboarded** that day. Not the running total of locked pledge. |
| `day_renewed_pledge` | flow | FIL/day | New pledge locked for sectors **renewed** that day. |
| `day_pledge_per_QAP` | cost | FIL/TiB QAP | Initial pledge required per TiB of QAP for sectors onboarded that day. A per-unit capital cost, not a flow. |
| `day_rewards_per_sector` | flow | FIL/day/sector | Daily block reward per 32 GiB sector (QAP-weighted). |
| `1y_return_per_sector` | flow | FIL/sector | 365-day rolling sum of `day_rewards_per_sector`. Absolute return, not normalized by pledge. **Array is 90 entries shorter than `forecast_length_days`.** |
| `1y_sector_roi` | ratio | — | `1y_return_per_sector / day_pledge_per_QAP`. Annual ROI (0.15 = 15%). **Array is 90 entries shorter than `forecast_length_days`.** |

### Vesting fields

All per-tranche vesting fields are **daily flows** (FIL/day) — they are zero once the vesting period ends. Most short-duration SAFT tranches are already exhausted in near-future forecasts.

| Field | Type | Unit | Description |
|---|---|---|---|
| `total_day_vest` | flow | FIL/day | Sum of all active vesting streams on that day. Read directly. |
| `total_vest` | cumstock | FIL | Cumulative total FIL vested from all sources since genesis. Diff to get daily (or use `total_day_vest`). |
| `six_month_vest_saft` | flow | FIL/day | 6-month SAFT tranche (~51.8M FIL total; exhausted ~Apr 2021 — zero in forecasts). |
| `one_year_vest_saft` | flow | FIL/day | 1-year SAFT tranche (~31.8M FIL total; exhausted ~Oct 2021 — zero in forecasts). |
| `two_year_vest_saft` | flow | FIL/day | 2-year SAFT tranche (~7.2M FIL total; exhausted ~Oct 2022 — zero in forecasts). |
| `three_year_vest_saft` | flow | FIL/day | 3-year SAFT tranche (~88.5M FIL total; exhausted ~Oct 2023 — zero in forecasts). |
| `six_year_vest_saft` | flow | FIL/day | 6-year SAFT tranche (~9.8M FIL total; exhausts ~Oct 2026). |
| `six_year_vest_pl` | flow | FIL/day | Protocol Labs 6-year linear vesting (300M FIL total, ~136 FIL/day; exhausts ~Oct 2026). |
| `six_year_vest_foundation` | flow | FIL/day | Filecoin Foundation 6-year vesting (100M FIL total, ~45 FIL/day; exhausts ~Oct 2026). |

### Index field

|---|---|---|---|

## Special Behaviors and Gotchas

- **`1y_sector_roi` and `1y_return_per_sector` are shorter than the forecast**: These arrays are **90 entries shorter** than `forecast_length_days` due to the 365-day rolling window convolution. The last valid entry corresponds to 90 days before the forecast end date. Do not align their tail with the forecast end.
- **`days` is not a forecast offset**: It counts from Filecoin network genesis (Oct 15, 2020). For example, if the simulation starts around Mar 2026, the first `days` value is approximately 2000, not 0.
- **`disbursed_reserve` is a constant broadcast**: Every element of the array is identical (~17.07M FIL). It is not time-varying. Do not compute trends or differences from it.
- **`rb_sched_expire_power_pib` and `qa_sched_expire_power_pib` are forecast-only**: Unlike most arrays that span historical window + forecast, these start at `current_date`. Their index 0 corresponds to tomorrow, not to the historical start.
- **`network_gas_burn` is a cumulative stock, not a daily rate**: Its value on any given day is the total FIL burned since genesis. To get a daily burn rate, diff consecutive Monday values and divide by 7.
- **Monday sampling and differences**: All `/simulate` arrays are Monday-sampled — consecutive elements are 7 days apart. When differencing a cumulative field, divide the result by 7 to get a per-day average.
- **Duplicate field pairs**: The following pairs are identical arrays — prefer the first name listed: `network_RBP_EIB` / `rb_total_power_eib`; `network_QAP_EIB` / `qa_total_power_eib`; `qa_day_onboarded_power_pib` / `day_onboarded_power_QAP_PIB`; `qa_day_renewed_power_pib` / `day_renewed_power_QAP_PIB`.
